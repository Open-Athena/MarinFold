# exp: fine-tune on regenerated only-correct docs vs. re-epoch original data

**Issue:** [#120](https://github.com/Open-Athena/MarinFold/issues/120) · **Kind:** `models` · **Branch:** `claude/exp120-regen-vs-reepoch`

## Question

Does continue-training Eric's tuned contacts-v1 1.5B on the **regenerated
"only-correct" documents** from [#100](https://github.com/Open-Athena/MarinFold/issues/100)
produce a better contacts-v1 model than **re-epoching the original** contacts-v1
training data, at matched training budget? This is the comparison #100 was built
to enable.

## Hypothesis

The regenerated docs are 100%-correct by construction and use the base model's
*preferred* contact ordering (lowest unmodified NLL), so they should be a
cleaner, higher-signal target than the originals. Expectation: the regenerated
arm beats the re-epoch arm on downstream contact prediction at the same token
budget. Counter-hypothesis: the regenerated set is low-entropy and on-policy, so
it may give little *new* signal and overfit quickly — hence the small epoch count
and reduced LR.

## Background

- **Base model** — Eric's #75 sweep winner (`prot-exp75-cv1-1_5b-e8-lr1e-3-wd0p2-v1-bc3084`,
  Qwen3 1.47B, eval loss **2.7566** at step 35679). Levanter checkpoint:
  `gs://marin-us-east5/checkpoints/prot-exp75-…-bc3084/checkpoints/step-35679`.
  Original run: 8 epochs at LR 1e-3, **batch 128**. (See exp89 for the exact
  Qwen3 config + a full downstream eval of this checkpoint.)
- **Treatment data (Arm B)** — #100's **941,004** regenerated only-correct
  documents (round-0 train proteins; 24 of 941,028 excluded by the correctness
  gate), published to `hf://buckets/open-athena/MarinFold/data/contacts-v1-train-only-correct-exp100/regenerated_documents.parquet`.
- **Baseline data (Arm A)** — the **original** round-0 contacts-v1 documents for
  the *same* 941,004 proteins (exp53 corpus, `round==0`).
- **Method template** — this is a Qwen3 analog of exp85 (the #67 Llama
  warm-restart): same `default_train` + `initialize_from_checkpoint_path`
  continue-train path, same `ArrayExemplar` cache-reader safety wrapper.
- **Downstream eval** — exp89's settled pairwise + K=10-ensemble harness on the
  554-protein curated eval set (`score_eval_set.py` + `compute_metrics.py`).

## Approach

Continue-train Eric's checkpoint under two matched arms differing **only in the
training document content**:

- **Arm A (re-epoch):** original round-0 documents.
- **Arm B (regenerated):** exp100 only-correct documents, same proteins.

Controls held fixed across arms: base checkpoint, Qwen3 config, tokenizer,
optimizer + LR schedule, epoch count per matched run, seed handling, and the
held-out eval sets + protocol. Warm-start loads model weights only (fresh step-0
/ optimizer / LR schedule / data loader) — a continue-train, not a resume.

**Batch = 128** (== Eric's #75 training batch), so the swept LRs are directly
interpretable as "below his 1e-3" with no batch-scaling confound. LR sweep
**{1e-4, 3e-4}**. Two schedule modes per (arm, LR):

- **headline** — a dedicated **1-epoch cosine** run (peak → 0.1·peak over one
  epoch). Issue #120's primary comparison point.
- **curve** — a **4-epoch flat-LR** run (cosine with `min_lr_ratio=1.0` = peak
  held after warmup) with a permanent checkpoint at **each epoch boundary**, so
  the 1/2/3/4-epoch points are directly comparable (no cosine-phase confound) and
  we can watch the regenerated arm for overfitting/collapse across passes.

Both arms use the same `num_train_steps` per epoch count — their token counts are
identical (same proteins, same true-contact set, order-only difference), so the
budget is matched. Trains on TPU (v5p-8 @ us-east5-a; marin checkpoint-resume
covers preemptions).

### Evaluation

1. **Val NLL — three sets monitored every eval** (issue #120 + Tim's comment,
   both arms):
   - `eval/contacts-v1-val-full/loss` — full original val split (canonical;
     step-0 ≈ 2.7566 confirms the warm-start loaded).
   - `eval/contacts-v1-val-orig/loss` — original round-0 val (9,558 proteins).
   - `eval/contacts-v1-val-regen/loss` — **regenerated** round-0 val (same 9,558
     proteins; the matched partner of `-orig`). Generated by re-running the
     exp100 pipeline on the val split — see below.
2. **Downstream contact-prediction accuracy** — export each checkpoint to HF
   (`export_checkpoints.py`) and run exp89's pairwise + K=10-ensemble harness
   (AUC / R-precision / contacts@{L,L/2,L/5}) on the 554-protein eval set. Run
   the full downstream eval on the headline 1-epoch models (both arms, both LRs)
   and the 4-epoch endpoints; the val curves cover the in-between.

## Files

## Results

### Warm-start sanity (passed)

All runs start at first-step **loss ≈ 2.6–2.7** (== Eric's base 2.7566), confirming
the cross-repo Qwen3 warm-start loaded correctly and `pad_tokenizer_to_match_model`
handled the vocab (a broken load reads ~8). Trained on v6e-8/us-east5-b.

### Headline (1 epoch) — eval loss

Held-out eval loss at step ~1005 (base checkpoint = 2.7566):

| arm | LR | val-full (real held-out) | val-orig (round-0) | val-regen (on-policy) |
|---|---|---|---|---|
| **A — re-epoch original** | 3e-4 | **2.744** | **2.721** | 2.666 |
| **B — regenerated** | 3e-4 | 2.837 | 2.815 | **2.525** |
| **B — regenerated** | 1e-4 | 2.836 | 2.815 | **2.537** |
| A — re-epoch original | 1e-4 | _(re-running after 1 preemption)_ | | |

**Read:** on the **real held-out original val**, re-epoching the original data
nudges loss *down* (2.7566 → 2.744), while the regenerated arm regresses *up* to
~2.84. On the **regenerated val** it flips (regen arm 2.52 vs re-epoch 2.67) — the
regenerated model fits its own low-entropy, on-policy distribution but does worse
on the true objective. This is the issue's counter-hypothesis on eval loss. LR
(1e-4 vs 3e-4) barely matters for the regen arm.

**Caveat (decider still pending):** issue #120 warns loss and accuracy can diverge
(cf. #89). The downstream contact-prediction accuracy (exp89 harness) is what
determines "the better contacts-v1 model" and is not yet run.
